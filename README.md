## The Problem

DAO treasuries and strategy vaults typically fail in one of two ways:

1. **Human bottlenecks**: decisions are slow, inconsistent, and hard to reproduce.
2. **Opaque automation**: bots execute actions but provide no tamper-evident reasoning trail.

When funds are managed by automation, stakeholders need one guarantee:

> **Every proposal must be explainable, verifiable, and constrained by governance.**

---

## The Solution

NeuroVault introduces a **stateless AI treasury agent** with auditable lifecycle controls:

- The agent rebuilds context from chain + configured data each cycle (no hidden memory state).
- It produces structured strategy output with confidence.
- Output is validated against safety constraints before any proposal path continues.
- Reasoning artifacts are committed to verifiable storage and linked to on-chain proposals.
- Final authority remains with governance voters.

✅ ENS identity is supported as optional metadata.  
✅ Agent access control is wallet/signature + role based (ENS is not required).

## How It Works

### Core Flow

1. Agent is triggered (`cron`, event, manual, or signed external request).
2. It assembles full context from on-chain state + configured feeds.
3. Gemini receives constraints + context and returns structured JSON.
4. Output is schema/safety validated.
5. Reasoning blob is committed (CID abstraction in current implementation).
6. Proposal payload is submitted to vault contract.
7. Stakers vote; only approved actions are executed.

---

## Reasoning Cycle (7 Stages)

NeuroVault implements a strict cycle model:

1. **Wake + deduplicate**  
   In-memory lock + cooldown gate. Concurrent trigger attempts are dropped.

2. **Context assembly**  
   Treasury balances, active goals, recent proposals, configured limits, approved target list, and yield feed context.

3. **LLM reasoning call**  
   Structured system prompt + dynamic context message.

4. **Output validation**  
   Checks action allowlist, amount sanity, spending limit, target allowlist, confidence threshold, and reasoning length.

5. **IPFS commitment**  
   Reasoning blob is serialized and committed via CID abstraction (deterministic fallback mode currently active).

6. **On-chain proposal submission**  
   Contract `propose(...)` is called with action params + CID.

7. **Cycle completion**  
   Lock release, timestamps update, and cycle result recorded.

> This stateless design makes runs reproducible from public + configured context.

## Trigger Sources

NeuroVault supports multiple trigger sources routed through the same dedup gate:

- **Cron scheduler**: heartbeat cycle (`CYCLE_INTERVAL`)  
- **Manual trigger**: `POST /cycle`  
- **External agent trigger**: `POST /propose` with wallet signature auth  
- **Typed event triggers**: `goal_updated`, `large_deposit`, `proposal_resolved` (endpoint-ready in current code path)

---

## Prompt Design

NeuroVault uses a two-part prompt strategy:

### 1) System Prompt (Invariant)
Defines:
- allowed actions,
- governance safety boundaries,
- required JSON output,
- confidence requirement,
- explainability requirements.

### 2) User Message (Dynamic)
Includes cycle context:
- treasury state,
- goals,
- recent proposal outcomes,
- trigger type,
- limit/allowlist configuration,
- external APY context if configured.

## Current Status: Implemented vs Planned

### ✅ Implemented

- 7-stage cycle orchestration in agent runtime

- strict output validation pre-submit
- signed external trigger endpoint (`POST /propose`)

## Getting Started

### Prerequisites

- Node.js 18+
- npm 9+

---

### 1) Clone

```bash
git clone https://github.com/minrawsjar/NeuroVault-Protocol.git
cd NeuroVault-Protocol
```

---

### 2) Frontend Setup

```bash
cd frontend
npm install
npm run dev
```

Open: `http://localhost:3000`

#### Frontend Env (`frontend/.env.local`)

```env
# Optional Gemini
GEMINI_API_KEY=

# Optional Lit authorized addresses (comma-separated)
LIT_AUTHORIZED_ADDRESSES=

# Vault source (production integration)
VAULT_DATA_SOURCE=http
VAULT_STATUS_URL=https://your-vault-api.example.com/status

# Optional local fallback for development only
ALLOW_LOCAL_FALLBACK=false
```

---

### 3) Agent Setup

```bash
cd ../agent
npm install
npm run dev
```

Default agent API: `http://localhost:3001`

#### Agent Env (`agent/.env`)

```env
GEMINI_API_KEY=
RPC_URL=
CONTRACT_ADDRESS=
AGENT_PRIVATE_KEY=

# Cycle config
CYCLE_INTERVAL=*/30 * * * *
CYCLE_COOLDOWN_MS=60000
MIN_CONFIDENCE=0.70

# Optional context feeds
BIFROST_APY_URL=
ACALA_APY_URL=

# Optional governance guardrails fallback
SPENDING_LIMIT_USD=0
APPROVED_TARGETS=

# Optional IPFS provider token
WEB3_STORAGE_TOKEN=

PORT=3001
```

## API Reference

### Agent API (`agent/src/index.ts`)

- `GET /status`  
  Service and cycle status

- `POST /cycle`  
  Manual trigger with optional `triggerType`

- `POST /propose`  
  External signed trigger

  Example body:

  ```json
  {
    "callerAddress": "0x...",
    "signature": "0x...",
    "timestamp": 1710000000000,
    "scenario": "Evaluate rebalance under volatility spike",
    "callerEns": "risk-agent.eth"
  }
  ```

- `GET /.well-known/skill.md`  
  Capability manifest for agent discovery workflows

### Frontend Bot API (`frontend/src/app/api/bot/route.ts`)

- `POST /api/bot`  
  Command runner for dashboard console (deterministic + fallback + advisory)

## Security Model

- **Defense in depth**:
  - pre-chain validation in agent,
  - on-chain constraints in vault contract,
  - governance approval before execution.

- **Access policy**:
  - wallet signature is used for external trigger auth,
  - ENS is optional identity metadata (not a gate).

- **Secret strategy**:
  - Lit primitives available for policy-gated secret handling workflows.

- **Auditability objective**:
  - reasoning artifacts are commit-linked and proposal-linked.

---

## Integration Notes

- Frontend is configured to use live integrations by default (`VAULT_DATA_SOURCE=http`).
- Local fallback data is disabled unless explicitly enabled with `ALLOW_LOCAL_FALLBACK=true`.
- ENS resolution is on-chain; ENS registration remains implementation-specific and should be wired to registrar transactions for production writes.
